# Networking — roadmap

The legacy callback/`old*` → modern async/`Result`/Swift-6 migration is **complete** (shipped across PRs #284–#303). This doc now tracks the next horizon: closing the gap between *modern* and *best-in-class*.

**Convention — keep the README in sync.** Any public-API change updates `README.md` in the **same** PR; its snippets are copy-paste docs, so a stale example is a broken one — and on the `actor` they must actually compile (isolated members need `await`). Run the suite with `make test` (local go-httpbin in Docker).

## Completed — the modernization

Condensed; full detail is in PRs #284–#303 and git history.

- **One typed async API.** Callback/`old*` verbs replaced by `get`/`post`/`put`/`patch`/`delete` → `Result<T, NetworkingError>` over any `Decodable` (or `JSONResponse` for status/headers, or `Data`). The entire `old*` + `JSONResult`/`NetworkingResult`/`Response` legacy hierarchy is deleted.
- **Downloads unified.** `downloadImage`/`downloadData` → `Result<T, NetworkingError>` where `T` is the bare payload (`Image`/`Data`) or an envelope (`ImageResponse`/`DataResponse`) carrying status/headers; `T` constrained by marker protocols.
- **Concurrency-safe.** `Networking` is a `public actor` (compiler-verified isolation; async config setters; no subclassing). Swift 6 language mode, `swift-tools 6.2`, iOS 18 minimum.
- **Deterministic tests.** Local `go-httpbin` (`make test`), zero `httpbin.org` references, CI on `macos-15` / Xcode 26.3.

These were breaking vs 7.0.0 → next tag is a major bump (see item 8 for the changelog/semver work).

## Roadmap — toward best-in-class

External review: the foundation is solid, but a top-tier Swift HTTP library would go further. Prioritized; 1–3 are the highest-leverage and add no dependency.

0. ~~**Fix README correctness.**~~ ✅ Stale `actor` examples (fake/config calls missing `await`) fixed so snippets compile.

1. ~~**Typed request bodies & builders — get off `Any`.**~~ ✅ The `parameters: Any?` + `ParameterType` pair and `fake*(response: Any?)` are gone, replaced by distinct labelled typed methods where the wrong combination can't be expressed (`Networking+HTTPRequests.swift`):
    - **Bodies** (`post`/`put`/`patch`): `body:` over any `Encodable` (JSON, `application/json`, ISO-8601 dates), `form:` over any flat `Encodable` (url-encoded; a `[String: String]` or your own model), `parts:`+`fields:` (multipart), `data:contentType:` (raw). Each returns `Result<T, NetworkingError>` or `Result<Void, …>`.
    - **Query** (`get`/`delete`): `query:` as a `[URLQueryItem]` (ordering/duplicate keys) or any flat `Encodable` model. `form:`/`query:` share `formFields` (`Networking+FormEncoding.swift`), which flattens via a JSON bridge + scalar re-decode so `Bool` stringifies to "true", not NSNumber's "1".
    - **Fakes:** `fake*(response:)` takes any `Encodable` (encoded to JSON), plus a no-body overload and the existing `fileName:` variant; `FakeRequest` stores a typed `Payload` enum. Internals (`handle`/`createRequest`/`httpBody`, `RequestBody`, `URLRequest` init `contentType:`, `logError`) carry no `Any`.
    - Header *values* were already `[String: String]`; a typed header-*name* vocabulary (enum of well-known keys) is a possible future ergonomics nicety, not an `Any` fix. The only remaining `Any` is the download cache substrate (`NSCache<AnyObject, AnyObject>`) and `AnyCodable` response bodies — both out of scope here.

2. ~~**Richer error model.**~~ ✅ `NetworkingError` (`NetworkingError.swift`) is now categorized by where the failure happened, preserving the underlying cause:
    - `invalidRequest(InvalidRequestReason)` (bad URL / un-encodable body or params), `transport(URLError)`, `http(HTTPError)`, `decoding(DecodingError, ResponseMetadata)`, `invalidResponse`, `cancelled`.
    - `HTTPError` carries `statusCode`, `isClientError`/`isServerError`, and `ResponseMetadata`. `decoding` keeps the live `DecodingError` (Sendable in this SDK) + metadata.
    - Cross-cutting conveniences: `statusCode`, `responseMetadata`, and a conservative `isRetryable` (transient transport failures + HTTP 408/429/5xx only).
    - The old flat `clientError`/`serverError`/`unexpectedError`/`invalidURL` cases and the `ParameterType`-era `[String: any Sendable]` details bag are gone.
    - **The core makes no assumption about the error body's shape.** `ResponseMetadata` retains the **full** `body: Data` (with `bodySnippet` a derived truncated log excerpt and `decode(_:)` a convenience), so a caller decodes its own structured error envelope into a typed value. The old in-core `ErrorResponse` parse and `HTTPError.serverMessage` are removed — body interpretation is the caller's (or an interceptor's) concern, not the core's. This subsumes any "parse Rails/JSON:API errors" feature: it's a few lines on the consumer side (`metadata.decode(MyError.self)`), not baked-in backend-specific logic.

3. ~~**Structured observability — remove `print`.**~~ ✅ The `print`-based `logError` (and `isErrorLoggingEnabled`/`setErrorLoggingEnabled`) are gone. Two cleanly-separated concerns (`NetworkingEvent.swift`):
    - **Event stream — the one consumer hook.** `events()` returns an `AsyncStream<NetworkingEvent>` that emits `.started(RequestContext)` then `.completed(RequestContext, outcome:duration:metrics:)` for *every* request (verbs and downloads). Multi-consumer; iterate with `for await` and accumulate into plain local state (no `Box`/`@unchecked`). Plumbing (continuations registry + single `emit` fan-out + `onTermination` cleanup + bounded `.bufferingNewest`) is ~15 lines on the actor. `RequestContext` = request-id `UUID` + method + url + the real request headers (raw; redaction is a log-path concern); `.completed` carries `Outcome` (`.success(statusCode:byteCount:)`/`.failure(NetworkingError)`), a measured `Duration`, and `TransactionMetrics` (a `Sendable` distillation of `URLSessionTaskMetrics`) for real network requests.
    - **Built-in logging — out of the box, synchronous/lossless** (in the completion funnel, not a stream consumer). `logLevel` (`Networking.LogLevel`) chooses *which* requests: `.none` / `.failures` (default) / `.all`; logged requests always get full detail — line + request & response headers + request & response bodies (truncated). `.failures` covers the debug case for free (incl. the request body, which catches wrong-shaped payloads); `.all` adds successes (with response body for "succeeded but wrong"). Downloads log line + request headers only (binary payload). **One redaction rule — debug shows, release redacts** (`redactsLogs`, defaulted via `#if DEBUG`): release logs replace both the body lines and the `setRedactedHeaderFields` header values with `<redacted>`. Redaction lives in the log path, so `events()` carries the real headers; it doesn't cover the parsed server message on the failure line.
    - **File sink for CLI/agent runs** (since `os.Logger` isn't in stdout — and `OSLogStore` can't read prior runs on iOS): `setLogFileURL(_:)` — or the `NETWORKING_LOG_FILE` env var (zero code change) — mirrors the same diagnostics to a readable, timestamped text file. A bare filename resolves under Caches (sandbox-safe; CocoaLumberjack's default), so it works in a simulator app too (`SIMCTL_CHILD_NETWORKING_LOG_FILE` + `simctl get_app_container`).

   **3.1 — Observability follow-up (done):** ✅ `events()` `AsyncStream` (the hook) · ✅ scope-based `logLevel` (`.none`/`.failures`/`.all`) · ✅ download-path events (`handleDataRequest`/`handleImageRequest`) · ✅ **simplified to one hook** — dropped the closure `setObserver` (it forced the `Box` dance and duplicated the stream); built-in logging runs in the completion funnel, defaults to failures. Later (own item if pursued): optional `os_signpost` intervals for Instruments.

4. **Request/response middleware.** A composable pipeline around every request — table stakes for a serious client. A short PR series; the seam shape was settled with a prior-art scan (chose the swift-openapi `ClientMiddleware` async-`next` onion over Alamofire's two-protocol adapt/retry split — one concept covers adapt + retry + auth).
    - ~~**4a — Interceptor seam (foundation).**~~ ✅ `HTTPInterceptor.intercept(_:next:)` (`HTTPInterceptor.swift`), wrapping every verb request **below** the generic decode layer (operates on a raw `HTTPExchange` = `Data` + `HTTPURLResponse`). Registered outermost-first via `setInterceptors`; folded around the real `URLSession` call in `perform` (`Networking+New.swift`). Calling `next` again replays the request.
    - ~~**4b — Auth-refresh, and the removal of `unauthorizedRequestCallback`.**~~ ✅ `AuthRefreshInterceptor`: on 401/403 it refreshes the credential and **replays** once (a fire-and-forget callback could only notify, never retry). Concurrent 401s share one refresh via a `RefreshCoordinator` actor (single-flight — the OkHttp stampede). `unauthorizedRequestCallback`/`setUnauthorizedRequestCallback` and the test-only `Box<T>: @unchecked Sendable` are deleted. Pure "notify me on 401" stays on `events()` (`.completed(_, .failure)`, `error.statusCode == 401`). **Downloads not yet routed through interceptors** (separate cache/task path) — fold in with #4c.
    - ~~**4c — Retry policy.**~~ ✅ `RetryInterceptor`: exponential backoff + full jitter, `maxAttempts` cap, `Retry-After` support (seconds or HTTP-date), driven by the `isRetryable` classifier from #2 (transient transport + `retryableStatusCodes`). Retryability predicates extracted onto `NetworkingError` as the single source of truth. **Idempotent methods only by default** (`retryableMethods`) so a retry can't duplicate a POST/PATCH side effect. **Downloads now route through the chain** (`requestData` → `perform`), so retry/auth apply to them too.
    - ~~**4d — Timeout policy.**~~ ✅ Already delivered by the existing `init(configuration:)` param: the `URLSessionConfiguration` is passed to `URLSession` untouched and the request init never overrides `timeoutInterval`, so `timeoutIntervalForRequest` / `timeoutIntervalForResource` govern — surfacing as `.transport(URLError(.timedOut))` (already retryable). No new API earned. (Per-request `timeout:` on every verb = API bloat, skipped; an overall-deadline-across-retries interceptor is the only net-new option, deferred until a real need appears.)
    - ~~**4e — Response validators.**~~ ✅ `ResponseValidatorInterceptor`: a `(HTTPExchange) -> .valid/.invalid(reason:)` check on **2xx** responses (content-type, envelope shape) — a rejection becomes `NetworkingError.validation(reason:metadata:)`. Non-2xx flows through unchanged so it stays `.http`. `ResponseMetadata(response:body:)` extracted as the single source for metadata.

5. **Optional `swift-http-types` substrate.** Apple's [`swift-http-types`](https://github.com/apple/swift-http-types) (version-independent `HTTPRequest`/`HTTPResponse`/`HTTPFields`; v1.6.0, Jun 2026) is now the standard low-level Swift HTTP currency. Add a lower-level request API built on it as a **separate SPM product**, so the core stays dependency-free.
    - **Consider a low-level escape hatch here** (e.g. `request(method:path:body:contentType:)`) for oddball method+body combinations the opinionated verbs deliberately don't expose — e.g. `DELETE` with a JSON body (real for bulk-delete / Elasticsearch-style APIs). One escape lets the caller own the non-standard case instead of growing per-method body overloads. `GET` with a body stays unsupported on purpose (RFC 9110: undefined semantics, and intermediaries/URLSession strip or reject it). Origin: PR #264.

6. **OpenAPI interoperability.** Apple's [Swift OpenAPI Generator](https://github.com/apple/swift-openapi-generator) is the type-safe-from-spec path (3.0/3.1, streaming, multipart, `OpenAPIURLSession` transport). Either offer an interop/transport example, or document clearly where Networking sits relative to it (the hand-written-client tier).

7. ~~**HTTP cache semantics (evaluate).**~~ ✅ **Decision: keep the bespoke cache, don't adopt `URLCache`.** A spike confirmed `URLCache` can't replace it without breaking functionality and public API: (a) the responses people cache (images/binaries) typically send **no** cache headers, so HTTP-semantics caching would store nothing and silently re-fetch — the bespoke cache's whole job is *unconditional* by-key caching; (b) `destinationURL(for:cacheName:)`, `imageFromCache(_:cacheName:)`, the `init(cache: NSCache)` param, and `cacheName:`/per-request `cachingLevel` have no `URLCache` equivalent. They're different tools (an explicit key→blob/image store vs. an HTTP-response freshness cache). Documented in the README's *Caching* section; consumers wanting true HTTP caching configure `URLSessionConfiguration.urlCache` via `init(configuration:)` (composes on the verb path).

8. **Package polish & release.** ✅ **CHANGELOG + semver notes** (`CHANGELOG.md`, v8 major bump from 7.0.0) and a real README *Installing* section (SPM + platform requirements) landed. Remaining (deferred, cluster around DocC): DocC docs + Swift Package Index `.spi.yml`, an examples app/package, and a compatibility/benchmark matrix — add once the v8 tag is cut and there's appetite (per the iOS convention: DocC comes late and stays gated).

9. **Follow-ups.**
    - **Error-body interpretation stays consumer-side — revisit only with a consumer.** The core hands back the full body (`ResponseMetadata.body` + `decode(_:)`) and makes no assumption about its shape — no `serverMessage`, no in-core parser, no Rails-specific anything. If a real consumer ever needs *structured* errors surfaced through `NetworkingError` (e.g. field-level validation for form UIs), add a general `middleware(Error)` seam so an interceptor can throw its own typed error — but build it only with that consumer in hand, not speculatively.

10. **Cache lifecycle — consistency + warm/cold TTL.** **← next.** The bespoke cache (in-memory `NSCache` + on-disk files under the networking domain folder) has two problems: cleanup is inconsistent, and disk files live forever. **One PR — it's all the same cache surface.**
    - **Consistency / latent bug.** `reset()` clears memory + disk (+ credentials); the static `Networking.deleteCachedFiles()` clears **only disk** — so after it, memory keeps serving "deleted" data until pressure eviction. A static method structurally can't touch a live instance's `NSCache`. Fix: cleanup goes instance-based and symmetric — `clearCache()` clears **both** tiers; `reset()` becomes `clearCache()` + wipe credentials/headers/fakes. Drop the public disk-only static (or keep a deprecated shim).
    - **Warm / cold expiry (the point is the *files*).** Today disk entries are unconditional and permanent — the cache only grows. Add a sliding TTL whose clock is **last use**:
        - **Warm** = used within `cacheTTL` → stays. Re-reading refreshes the clock, so an entry in active use never goes cold while the process lives.
        - **Cold** = idle beyond `cacheTTL` → swept: lazily when re-requested, and in bulk on a sweep at startup (off the request path).
        - **Tracking is cheap and in-memory** — an actor-held `[key: Date]` last-access map, updated on each hit. **Never persisted, never written to the files** (no per-read I/O, no manifest to keep in sync). Entries with no in-session record (e.g. right after launch) fall back to the file's own creation date.
        - **Accepted limitation:** access times aren't persisted, so warmth resets across launches — at launch an entry is judged by its file date. That's the deliberate price of "no manifest, no file writes"; it still bounds disk growth.
    - **`cacheTTL: Duration` — non-optional, default 7 days,** modifiable (init param + actor-isolated setter). No `nil` and no "never expire" flag; there's always a TTL (use a long duration if you must).
    - **Memory tier stays hands-off — expose no `NSCache` limits.** `countLimit` is object *count*, `totalCostLimit` is in caller-defined "cost" units (meaningless unless we pass byte costs to `setObject`, which we don't) — both are confusing and unguessable. `NSCache` already evicts under memory pressure; let it.
    - **No per-entry public removal** (`removeCachedResponse(for:)`) — no consumer has asked; don't build it speculatively.
    - **Folds in the 255-char filename fix** (same surface): a long URL becomes one filename component over the filesystem's 255-char cap and the write throws (pinned by `testCachingALongURLFailsToWriteFile`). Hash the component when it's too long (readable prefix + hash suffix so distinct URLs can't collide); flip the test to assert success.
